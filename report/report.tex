\section{Introduction}

\subsection{Motivation}

Large language models (LLMs) have seen broad adoption across nearly every sector, assisting with tasks ranging from natural language understanding to complex reasoning and software development~\cite{chang2024survey}. One of their most impactful applications is code generation, where they help programmers by automating development tasks and reducing human error~\cite{llmgen2025}. While most existing benchmarks such as HumanEval~\cite{chen2021evaluatinglargelanguagemodels} and MBPP~\cite{austin2021programsynthesislargelanguage} focus primarily on correctness and functionality, the performance of written code in terms of memory footprint and runtime efficiency is largely overlooked. However, in scientific computing and graph analysis small algorithmic can result in a reduction of runtime and energy consumption by orders of magnitude~\cite{llmgen2025}.

\citeauthor{llmgen2025}~\cite{llmgen2025} addressed this gap by benchmarking eight state-of-the art LLMs on their ability to generate efficient C implementations of algorithms for graph analysis. To reduce the risk of models simply reproducing memorized training data, each LLM is supplied with the C source files of an existing benchmark suite~\cite{bader2023tc} and require the generate code to integrate into it. Their results shows that contemporary LLMs can produce generative implementations, often surpassing human-written baselines~\cite{llmgen2025}.


\subsection{Cross-Language Code Generation}

A natural extension of their work is to evaluate LLMs when the target language differs from the language of the provided context~\cite{llmgen2025}. This scenario of cross-language code generation is practically relevant, as many legacy systems are running on lower-level languages such as C or COBAL, while recent software development targets higher-level languages such as Java or Python~\cite{cobol2000}. In this setup, the LLM must have a thorough understanding of the underlying algorithms and data structures in order to produce a valid implementation independently, without relying on the provided context language.

\subsection{Contribution}

This work extends the evaluation of \citeauthor{llmgen2025} under the cross-language setting described in the previous section. Three state-of-the-art LLMs are prompted to generate Java implementations of graph algorithms while only C code is provided as context. It is important to note that no Java infrastructure is supplied. Each model must imagine how the C benchmark suite would look when translated to Java and generate code that integrates into this framework. This makes the evaluation strictly harder than same-language generation.

Similar to~\cite{llmgen2025}, two distinct approaches are evaluated. First, an optimization approach for triangle counting that evaluates the performance of LLMs to outperform existing benchmarks. Second, an algorithm synthesis approach covering triangle counting, diameter finding, and clique number. Both approaches are evaluated using RMAT benchmark graphs~\cite{rmat2004} at scales 6 to 14. Each implementation is evaluated on compilability, correctness, execution time, and memory usage, following the methodology of the original paper. 

\subsection{Report Structure}

The remainder of this report is organized as follows. Section~\ref{sec:testing} describes the experimental setup, the evaluated LLMs, the used prompts. Additionally, the obtained numerical results are reported and compared with the results from the original paper~\cite{llmgen2025}. Section~\ref{sec:conclusions} summarizes our findings, discusses the implications of cross-language code generation, and suggests directions for future work.

\section{Numerical Testing}
\label{sec:testing}

\subsection{Testing Environment}

All experiments were conducted on an Apple MacBook Pro with an Apple M3 chip featuring 4 performance cores running at up to 4.05 GHz and 4 efficiency cores running at up to 2.75 GHz, with 16 GB of unified DRAM. The performance cores feature 192 KB of L1 instruction cache, 128 KB of L1 data cache, and a shared 16 MB L2 cache. The efficiency cores feature 128 KB of L1 instruction cache, 64 KB of L1 data cache, and a shared 4 MB L2 cache. The cache line size is 128 bytes. Java implementations were compiled and executed using Java 21 (Hotspot JVM, build 21.0.10). Each algorithm is averaged over 10 runs to smooth out measurement noise, consistent with the methodology of the original paper~\cite{llmgen2025}. Note that absolute runtime results are not comparable with the original study which used an AMD EPYC 7753 server at 2.45 GHz with 512 GB of RAM~\cite{llmgen2025}.

\textbf{Anthropic Claude Opus 4.6:} Anthropic's Claude models are renowned for their advanced reasoning, language comprehension, and efficient code generation~\cite{llmgen2025}. We selected Claude Opus 4.6, released on February 5, 2026, which is currently Anthropic's most capable model. Compared to Claude Sonnet 4 which was used in the original paper, Opus 4.6 is specifically designed for superior coding, complex reasoning, and agentic workflow integration. Featuring a context window of up to 1 million tokens, it achieves state-of-the-art performance on coding benchmarks such as Terminal-Bench 2.0~\cite{anthropic2026opus46}. We therefore expect it to produce more sophisticated and efficient algorithm implementations than its Sonnet predecessor. 

\textbf{Google Gemini 3.1 Pro:} In recent years, Google has gained significant attention through the release of several highly capable multimodal models, thereby positioning Gemini as a leading model family for text, images, and code~\cite{google2026gemini31pro}. In the original paper, the Gemini 2.5 Flash model was not able to keep up with the more advanced models from Anthropic and OpenAI, though Gemini 2.5 Pro remained competitive and achieved respectable efficiency rates and runtimes. The release of Gemini 3.1 Pro on February 19, 2026, represented a substantial improvement over its predecessors, achieving more than double the reasoning performance on competitive coding benchmarks~\cite{google2026gemini31pro}, making it a compelling candidate for our cross-language benchmark.

\textbf{OpenAI GPT-5.3-Codex:} With the release of ChatGPT in 2022, OpenAI is widely regarded as one of the pioneers of the modern era of generative AI that has fundamentally changed the public perception of the capabilities of large language models~\cite{llmgen2025}. Its Codex line specifically refers to a family of models built for code generation and software engineering tasks. The GPT-5.3-Codex model was released on February 5, 2026, and advances coding performance and reasoning capabilities, showing strong performance on modern benchmarks. For our experiments, the model was accessed via the GitHub Copilot web interface.
